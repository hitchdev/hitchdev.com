Why not use the page object pattern?

The page object pattern is a design pattern for creating a code API abstraction around web pages to simplify their usage in GUI tests.

The general idea is to enact a separation of concerns between providing an API around HTML and code that wrapping and providing an API around the web application.

  1. It leads to an overabundance of test code where configuration would have been sufficient ——————————————————————————————-

The thorniest complexity in interacting with web pages is not really about what the user does, it’s about finding elements on the page.

What is often obvious to the user (e.g. username text box) is often ambiguous to code. This is, in part, because HTML’s design did not make this easy. However, HTML is here and we’re stuck with it so we need to deal with it.

Selenium has a number of different kinds of selectors (ID, class, attribute, xpath) each of which is appropriate to use in multitude of different situations. None of this is remotely interesting to anybody except the person writing the test - users just want to grab elements and perform actions upon them. Whether an element is grabbed by a class selector or an ID selector is of no importance.

Separating the concern of “how to grab an element” from “what to do with it”

If the problem of finding elements on a page can be cleanly abstracted, the problem of what to do with them doesn’t have to be abstracted.

The user might click, enter text, drag and drop, scroll to, etc. Whatever you do with an element that is part of the specification - a story in a todo app where a “todo” is created by dragging and dropping an existing todo on to a “clone” space is substantially different to a story where a todo is created by entering text and clicking a button.

If the user actions on elements are exposed at the highest level in a test that removes ambiguity about what the test actually does is better than a test that leaves user interactions ambiguous.

  1. It often leads to relevant details about the test’s interaction with the web page being concealed in code rather than exposed ——————————————————————————————————————————–

A clear example of this anti-pattern can be seen in this tutorial, ironically titled “how to write better acceptance tests”.

The page object for “Todo” has a step “create”. How is the todo created though? What did the mock user do while they were creating? Can you tell from the scenario? Nope. You have to dig in to the code in order to do figure out what happened. It literally just gives you the most abstract view possible, which would not be enough for the “customer” (say, the CEO of this todo-app business) to know if it did what they wanted.

Of course, this is still a tendency and it is entirely possible to create page objects that do not do this. Sadly, this is a tendency that is frequently actively encouraged as a “best practice”.
